FE 개발일기 [4] 체계 잡기

Danga Lee
2024-03-10

Goal 💡

  • Git 커밋 메시지
  • Git-flow, 브랜치 전략

[1] Git message

코딩 컨벤션과 마찬가지로 개발자 간 커밋 메시지를 통일했습니다. 일관된 형식의 커밋 메시지를 활용하면 메시지의 가독성이 높아지고 문제 발생 시 이력을 추적하기가 용이합니다.
통일감을 주는 것에는 성공했지만, 실제 개발은 아직까지 시간 내에 기능을 개발하는 데 급급한 나머지 Git 커밋 메시지를 잘 활용하지 못했습니다.

### [이슈번호] 커밋타입 : 최대한 자세히 제목 작성(괄호 및 띄어쓰기 준수하기)
### ex) [S10P12A401-71] feat : 리프레시 토큰 jpa로 변경

##### 본문

# - 1. 무엇을 수정했는지 ex) - 1. 리프레시 컨트롤러 수정 
# - 2. 왜 수정했는지 ex) - 2. 엔티티 수를 줄이고, 시큐리티 강화를 위해
- 1.
- 2.

#   [커밋 타입]  리스트
#   feat      : 기능 (새로운 기능)
#   fix       : 버그 (버그 수정)
#   refactor  : 리팩토링
#   style     : 스타일 (코드 형식, 세미콜론 추가: 비즈니스 로직에 변경 없음)
#   docs      : 문서 (문서 추가, 수정, 삭제)
#   test      : 테스트 (테스트 코드 추가, 수정, 삭제: 비즈니스 로직에 변경 없음)
#   chore     : 기타 변경사항 (빌드 스크립트 수정 등)

# ------------------
#   [체크리스트]
#     커밋타입을 소문자로 작성했나요?
#     제목은 명령문으로 작성했나요?
#     제목 끝에 마침표(.) 금지
#     제목과 본문을 한 줄 띄워 분리하기
#     본문에 여러줄의 메시지를 작성할 땐 "-"로 구분했나요?
# ------------------

[2] Git-flow, 브랜치 전략

6인 프로젝트를 진행하며 처음으로 브랜치를 나누어 작업했습니다.

master 브랜치: 최종 프론트엔드 + 백엔드 코드를 모아두는 브랜치입니다.

frontend-deploy 브랜치/backend-deploy 브랜치: 해당 브랜치에 merge된 소스를 통해 pipeline이 동작합니다.

frontend 브랜치, backend브랜치: 하위에 기능 구현 별 여러 feature 브랜치를 두고 주기적으로 frontend/backend 브랜치에 merge했습니다.
